# Flutter Forms & User Input

Flutter provides two complementary approaches to collecting text from users: `TextField` widget and the higher-level `Form` + `TextFormField` combination. This section focuses on the `Form` + `TextFormField` system because it gives you built-in validation coordination with zero manual wiring.

## TextField vs. TextFormField

| Widget          | Best for                                                                       |
| :-------------- | :----------------------------------------------------------------------------- |
| `TextField`     | A standalone input where you handle validation yourself                        |
| `TextFormField` | An input that lives inside a `Form` and participates in coordinated validation |

Both widgets look the same to the user. The difference is entirely in how you manage their values and validation lifecycle.

## TextEditingController

A `TextEditingController` is a mutable object that tracks the current text, selection, and cursor position of an input widget. You attach it to a `TextFormField` and read from or write to it from outside the widget tree:

```dart
class _ProductFormState extends ConsumerState<ProductForm> {
  late final TextEditingController _nameController;

  @override
  void initState() {
    super.initState();
    // Pre-populate the field if editing an existing product
    _nameController = TextEditingController(text: widget.product?.name);
  }

  @override
  void dispose() {
    _nameController.dispose(); // Always dispose to avoid memory leaks
    super.dispose();
  }
}
```

> **Important:** Controllers must be disposed when the widget is removed from the tree. Failing to dispose leads to memory leaks because the controller holds listeners that prevent garbage collection.

Controllers are typically declared `late final`: `late` because they are initialised in `initState` rather than at declaration time, and `final` because the controller instance itself never changes, only the text value it tracks changes.

### Reading and Writing Values

```dart
// Read the current text value
final name = _nameController.text;

// Set a value programmatically
_nameController.text = 'New value';
```

In the form submit handler, you typically read values, trim whitespace, and parse into the types your domain layer expects:

```dart
Future<void> _saveForm() async {
  final name = _nameController.text.trim();
  final amount = double.tryParse(_priceController.text.trim()) ?? 0.0;
  final discount = double.tryParse(_discountController.text.trim());
}
```

## The Form Widget

`Form` is a container widget that coordinates validation across all `TextFormField` children at once. It requires a `GlobalKey<FormState>` to allow code outside the widget tree to trigger validation programmatically:

```dart
class _ProductFormState extends ConsumerState<ProductForm> {
  final _formKey = GlobalKey<FormState>();

  @override
  Widget build(BuildContext context) {
    return Form(
      key: _formKey,
      child: Column(
        children: [
          TextFormField(/* ... */),
          TextFormField(/* ... */),
        ],
      ),
    );
  }
}
```

`GlobalKey<FormState>` gives you a unique handle to the `FormState` object. This key must be declared as a `final` field on the `State` class and should never be created inside `build()`, because that would create a new key on every rebuild and break the connection to the previous form state.

### Triggering Validation

When the user taps submit, call `validate()` on the `FormState` to trigger every child field's validator simultaneously:

```dart
Future<void> _saveForm() async {
  if (!(_formKey.currentState?.validate() ?? false)) return;
  // All validators returned null; safe to proceed
}
```

`validate()` returns `true` only when **every** field's validator returns `null`. If any validator returns an error string, that string appears beneath the offending field and `validate()` returns `false`.

### Reset Validation

```dart
void _resetForm() {
  _formKey.currentState?.reset();
}
```

Whenever we need to clear the validation messages for all fields, we can call `reset()` on the `FormState`. It resets all fields to their initial state.

## InputDecorationTheme

Rather than styling each `TextFormField` individually, the project defines a centralised `InputDecorationTheme` inside `AppTheme`. This keeps all form fields visually consistent with zero per-widget style code:

```dart
// core/theme/app_theme.dart
inputDecorationTheme: InputDecorationTheme(
  filled: true,
  isDense: true,
  fillColor: _onBackground.withAlpha(20),
  hintStyle: TextStyle(color: _onBackground.withAlpha(150)),
  contentPadding: const EdgeInsets.symmetric(
    vertical: AppDimensions.defaultPadding / 2,
    horizontal: AppDimensions.defaultPadding / 3,
  ),
  enabledBorder: OutlineInputBorder(
    borderRadius: BorderRadius.circular(AppDimensions.defaultBorderRadius),
    borderSide: BorderSide.none,
  ),
  focusedBorder: OutlineInputBorder(
    borderRadius: BorderRadius.circular(AppDimensions.defaultBorderRadius),
    borderSide: BorderSide(color: _primary),
  ),
  errorBorder: OutlineInputBorder(
    borderRadius: BorderRadius.circular(AppDimensions.defaultBorderRadius),
    borderSide: BorderSide(color: _secondary.withAlpha(200)),
  ),
  focusedErrorBorder: OutlineInputBorder(
    borderRadius: BorderRadius.circular(AppDimensions.defaultBorderRadius),
    borderSide: BorderSide(color: _secondary),
  ),
),
```

Four border states are defined explicitly:

| Border state         | When applied                                    |
| :------------------- | :---------------------------------------------- |
| `enabledBorder`      | Field is idle with no input focus and no error  |
| `focusedBorder`      | Field currently has keyboard focus              |
| `errorBorder`        | Field has a validation error and no focus       |
| `focusedErrorBorder` | Field has a validation error and keyboard focus |

Because all four states are set at theme level, individual `TextFormField` widgets only need to supply a `hintText` or `suffixIcon`; they inherit everything else automatically.

## Multiline Fields and Character Limits

Setting `maxLines` to a value greater than one creates a multiline input. Combining it with `maxLength` automatically adds a character counter below the field:

```dart
TextFormField(
  controller: descriptionController,
  decoration: const InputDecoration(hintText: 'Enter product description'),
  maxLines: 5,
  maxLength: 500,
  validator: _descriptionValidator,
)
```

## Keyboard Type

Setting `keyboardType` tells the OS which keyboard layout to show. For numeric inputs, use `TextInputType.numberWithOptions`:

```dart
TextFormField(
  keyboardType: const TextInputType.numberWithOptions(decimal: true),
  decoration: const InputDecoration(hintText: '0.00'),
)
```

Common `keyboardType` values:

| Value                                            | Keyboard shown                              |
| :----------------------------------------------- | :------------------------------------------ |
| `TextInputType.text`                             | Standard full keyboard (default)            |
| `TextInputType.number`                           | Numeric pad                                 |
| `TextInputType.numberWithOptions(decimal: true)` | Numeric pad with decimal point              |
| `TextInputType.emailAddress`                     | Standard keyboard optimised for email entry |
| `TextInputType.url`                              | Standard keyboard optimised for URL entry   |
